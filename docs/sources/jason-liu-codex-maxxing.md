## Why This Source Matters

Jason Liu's article is useful because it treats Codex as a place where work can live, not only as a tool for code edits. That framing fits the course refresh as long as it is translated for non-developers.

## Core Idea

The article argues that recent Codex app upgrades make broader work feel native: durable threads, memory, tools, steering, resumability, and artifact review combine into an operating loop.

Plain-English translation:

Codex is more useful when the work has a home, a memory, a review surface, and a way to keep moving after the first prompt.

## Concepts To Carry Into The Course

### Durable Threads

Jason describes pinned long-running threads for important workstreams. The course should translate this as: keep an ongoing thread when the work has continuity, but avoid mixing unrelated jobs.

Course use:

- Teach "one thread per workstream."
- Explain compaction as summarizing long work so the thread can continue.
- Mention that continuity has cost and clutter tradeoffs.

### Voice Input

Voice matters because it captures rough thinking that people may not type. For non-developers, this is a helpful permission slip: Codex can work with messy context if the learner gives it enough raw material.

Course use:

- Encourage learners to speak the messy version first.
- Then ask Codex to turn it into a work contract.

### Steering

The article frames steering as adding intent while work is in progress. The course should explain this as "keep shaping the task while Codex works."

Course use:

- Teach short steering messages: "make this smaller," "keep the format," "wait for feedback," "open a PR."
- Pair steering with review surfaces so feedback is concrete.

### Memory As Files

Jason's vault pattern is especially important: important work memory becomes inspectable files, not only a long transcript.

Course use:

- Teach `AGENTS.md`, project notes, TODOs, and decision logs as durable context.
- Explain first-party memories as useful recall, not a replacement for checked-in instructions.

### Browser And Computer Use

The article separates browser, Chrome, and computer use by what kind of surface Codex needs to operate.

Course use:

- In-app browser for local pages and visual review.
- Chrome for signed-in browser state.
- Computer Use for GUI-only workflows.

### Remote Control

Remote control matters when Codex keeps working while the learner changes location. For non-developers, this is less about infrastructure and more about preserving momentum.

Course use:

- Explain "start at desk, unblock from phone" as the core pattern.

### Heartbeats

Jason's examples show thread-local recurring loops: checking messages, feedback, or a support flow until something changes.

Course use:

- Teach heartbeats as "check back here later."
- Keep examples practical and safe: feedback monitoring, PR comments, doc comments, long-running renders.

### Goals

The article's key lesson is that ambition needs verification. A goal is strong when Codex can compare progress against a real oracle such as tests, a checklist, or an artifact.

Course use:

- Pair Goals with Chris Hayduk's measurable-goal structure.

### Side Panel And Artifacts

The side panel is framed as the place the work happens: inspect artifacts, operate web surfaces, and review changes.

Course use:

- Show that outputs are not only text responses.
- Teach lightweight `index.html` artifacts, Markdown notes, spreadsheets, PDFs, slides, and browser previews as reviewable work.

## Course Caution

Jason writes as an advanced power user. The course should keep his workflow as inspiration, not the baseline expectation for a beginner.

Plain-English version:

Start with one useful loop. Then make it durable only after it proves valuable.
